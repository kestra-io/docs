# Kestra documentation style guide

Use these rules when editing prose in the docs repo.

## Core principles

- Be clear before being clever.
- Optimize for task completion.
- Explain the why, not just the how.
- Prefer consistency over novelty.
- Write for a global technical audience.

## Voice and tone

- Write in second person when instructing the reader.
- Prefer active voice.
- Prefer present tense for facts and product behavior.
- Use a direct, calm, technically confident tone.
- Avoid hype, filler, and editorial throat-clearing.
- Avoid anthropomorphism.

## Page openings

The first sentence should:

- explain what the page covers
- stay concise
- avoid filler such as `In this page` or `This document will`
- stand on its own in cards, previews, and search results

## Headings

- Use sentence case.
- Make headings descriptive.
- Do not restate the title as the first H2.
- Keep heading depth shallow where possible.

## Procedures

- Introduce the goal before the steps.
- Include prerequisites before procedural steps.
- Use imperative verbs in numbered steps.
- End with verification or next steps.

## Code and examples

- Introduce code blocks so the reader knows what to look for.
- Explain the result or tradeoff after the block when helpful.
- Prefer complete examples in how-to guides unless omission is explicitly called out.
- Mark incomplete snippets clearly.

## Links and media

- Use descriptive link text.
- Link to the source of truth instead of duplicating content.
- Use screenshots only when they materially help the reader.
- Write descriptive alt text.

## What to remove

- Promotional language
- Vague claims such as `easy`, `powerful`, or `flexible` without context
- Long openings before the first useful detail
- Decorative screenshots
- Code blocks with no framing or explanation

## Review checklist

1. The page type is clear.
2. The first sentence works as standalone summary text.
3. Headings use sentence case.
4. Prose is direct and active.
5. Terminology is consistent.
6. Links and screenshots earn their place.
7. The reader can verify success or find the next relevant path.
